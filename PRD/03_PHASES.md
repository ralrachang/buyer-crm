## Phase 1: MVP (1~2주)

### 목표
패스워드로 접근해서 고객 등록·조회·타임라인 작성까지 실제로 쓸 수 있는 버전.

### 기능
- [ ] 패스워드 로그인 화면 (공유 패스워드 1개)
- [ ] 고객 목록 페이지 (@이름 전체 보기)
- [ ] 고객 등록 / 편집 / 삭제
- [ ] 9개 기본정보 저장 (이름, 매입목적, 매수의지, 보유현금, 선호지역, 고객컨셉, 직업, 나이, 법인보유)
- [ ] 고객 상세 페이지 (프로필 카드 + 타임라인)
- [ ] 타임라인 @멘션 + 내용 입력 → 피드 추가

### 데이터
- Customer, TimelineEntry

### 인증
- 환경변수에 저장된 공유 패스워드 비교. 세션 쿠키로 유지.

### "진짜 제품" 체크리스트
- [ ] 실제 Supabase DB 연결 (목업 데이터 X)
- [ ] 실제 패스워드 인증 (.env 환경변수 사용, 하드코딩 X)
- [ ] Vercel에 실제 배포 (localhost X)
- [ ] 팀원이 URL로 접속해서 써볼 수 있음

### Phase 1 시작 프롬프트
```
이 PRD를 읽고 Phase 1을 구현해주세요.
@PRD/01_PRD.md
@PRD/02_DATA_MODEL.md
@PRD/04_PROJECT_SPEC.md

Phase 1 범위:
- 패스워드 로그인
- 고객 목록 (CRUD)
- 고객 상세 + 타임라인 입력

반드시 지켜야 할 것:
- 04_PROJECT_SPEC.md의 "절대 하지 마" 목록 준수
- 실제 Supabase DB 연결 (목업 X)
- 패스워드는 .env.local의 SITE_PASSWORD 환경변수 사용
- 모바일 반응형 필수
```

## Phase 2: 확장 (Phase 1 완료 후 1~2주)

### 전제 조건
- Phase 1이 Vercel에 안정적으로 배포된 상태
- 실제 고객 데이터 10건 이상 축적

### 목표
목록에서 매수의지 레벨을 직관적으로 파악하고, 빠른 검색으로 고객 찾기.

### 기능
- [ ] 매수의지 레벨(1-5) 색상/배지 시각화 (목록 + 상세 페이지)
- [ ] 고객 목록 정렬 (레벨 높은 순 / 최근 업데이트 순)
- [ ] 타임라인/고객명 기반 검색

### 추가 데이터
- Customer.purchase_intent 필드 활용 (이미 Phase 1에서 저장됨, UI만 추가)

### 통합 테스트
- [ ] Phase 1 기능(로그인, 고객 CRUD, 타임라인)이 여전히 정상 동작하는지 확인

---

## Phase 3: 고도화 (Phase 2 완료 후)

### 전제 조건
- Phase 1 + 2가 안정적으로 운영 중

### 목표
고객별 매물 제안 이력을 체계화하여 "이 고객에게 어떤 매물을 언제 보여줬는지" 추적.

### 기능
- [ ] 매물 매칭 메모 기능 (PropertyMemo 테이블 추가)
- [ ] 고객별 매물 제안 이력 조회
- [ ] 다음 연락 예정일 알림 (팅 기능)

### 주의사항
- PropertyMemo 테이블 추가 시 Supabase 마이그레이션 필요
- 알림 기능은 외부 서비스(이메일/푸시) 연동 비용 발생 가능
